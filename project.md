# 无人机智能编排数据集生成方案

> 版本：v1.0 | 日期：2026-05 | 状态：规划中

---

## 一、项目背景

### 1.1 业务背景

现有无人机编排平台基于组件化架构，用户需要手动选择并拼接任务组件（如起飞、飞往、悬停、巡检等）来构建任务拓扑。这种方式对用户的专业知识要求较高，操作门槛大，效率低。

为提升平台智能化水平，计划新增**智能编排功能**：用户只需输入结构化的自然语言描述，系统自动生成合法的任务组件拓扑，大幅降低使用门槛，提升任务规划效率。

### 1.2 问题定义

| 项目 | 描述 |
|------|------|
| **输入** | 结构化文本（任务类型、地点、高度、天气、特殊要求等） |
| **输出** | 组件拓扑序列（起飞 → 爬升 → 飞往目标 → 巡检 → 返航 → 降落） |
| **任务类型** | Seq2Seq / Structured Prediction |
| **实现方式** | 基于预训练大语言模型微调 |
| **核心瓶颈** | 缺乏训练数据，需构建专属数据集 |

### 1.3 目标

- 生成 **2000~3000 条**高质量、覆盖多场景的训练数据
- 数据覆盖 **8~12 种**主要任务类型
- 验证集人工抽检准确率达到 **90% 以上**
- 最终支撑模型微调，实现智能编排功能上线

## 三、关键技术

### 3.1 方法一：模板引擎（Rule-Based Generation）

**原理**：人工定义每种任务类型的骨架拓扑 + 条件规则，结合参数空间全组合采样生成数据。

**适用场景**：快速生成大量结构正确、分布可控的基础数据。

**核心实现**：

```python
TASK_TEMPLATES = {
    "电力巡检": {
        # 骨架：所有场景都包含的基础序列
        "skeleton": [
            "起飞", "爬升至{飞行高度}", "飞往目标区域",
            "沿线巡检", "返航", "降落"
        ],
        # 条件规则：根据参数动态插入组件
        "conditionals": {
            "特殊要求=热成像": {
                "insert_before": "沿线巡检",
                "components": ["热成像开启"],
                "insert_after": "沿线巡检",
                "components_after": ["热成像关闭"]
            },
            "天气条件=夜间": {
                "insert_before": "起飞",
                "components": ["夜航灯开启"]
            }
        }
    }
}

# 参数空间：所有字段的可选值
PARAMS_SPACE = {
    "电力巡检": {
        "任务地点": ["110kV输电线路A段", "220kV变电站北区", "风电场1号"],
        "飞行高度": ["50m", "80m", "100m", "120m"],
        "天气条件": ["晴天", "阴天", "夜间"],
        "特殊要求": ["无", "热成像", "视频录制"]
    }
}
```

**优势**：速度快、结构100%合法、分布可控
**局限**：多样性有限，措辞单一

### 3.2 方法二：LLM 正向生成（Forward Generation）

**原理**：将组件库和约束规则注入 System Prompt，让 LLM 同时生成输入描述和对应的输出拓扑，批量调用生成多样化数据。

**适用场景**：生成语义丰富、措辞多样的数据，补充模板无法覆盖的边缘场景。

**Prompt 设计**：

```
[System]
你是无人机任务规划专家，熟悉以下组件库：
{component_library}

生成规则：
1. 拓扑必须以"起飞"开始，以"降落"结束
2. 只能使用组件库中的组件名称
3. 组件顺序必须符合实际飞行逻辑
4. 输出严格按 JSON 格式，不包含任何额外文字

[User]
请生成 20 条多样化的无人机任务数据：
- 任务类型覆盖：电力巡检、农业植保、搜救任务
- 包含简单场景（4~6个组件）和复杂场景（8~12个组件）
- 参数值要多样，避免重复

输出格式（JSON数组）：
[{"input": {...}, "output": "起飞 -> ... -> 降落"}]
```

**批量调用策略**：每批生成 20 条，共调用 50 批，目标生成 1000 条原始数据（过滤后约 800 条有效）。

---

### 3.3 方法三：LLM 反向生成（Reverse Generation）

**原理**：给定一条已知合法的拓扑（来自种子数据或模板生成），让 LLM 生成多条可以推导出该拓扑的不同任务描述，实现"一拓扑多描述"的数据扩增。

**适用场景**：增加描述措辞多样性，提升模型对不同表达方式的泛化能力。

```
[Prompt]
已知无人机执行了如下任务拓扑：
起飞 -> 爬升至100m -> 飞往目标区域 -> 热成像开启 -> 航线巡检 -> 热成像关闭 -> 返航 -> 降落

请生成 5 条不同的结构化任务描述，要求：
- 每条描述都能合理推导出上述拓扑
- 字段组合略有差异（有些详细，有些简略）
- 措辞自然多样

输出 JSON 数组：[{"任务类型": "...", "任务地点": "...", ...}]
```

**扩增效果**：100 条种子拓扑 × 5 条描述 = 500 条新数据。

### 3.4 数据校验模块

所有生成数据必须经过三层校验：

**第一层：结构校验（规则）**

```python
def validate_structure(item, component_set):
    components = [c.strip() for c in item["output"].split("->")]
    assert components[0] == "起飞",          "必须以起飞开始"
    assert components[-1] == "降落",         "必须以降落结束"
    assert len(components) >= 4,             "组件数不足"
    assert len(components) <= 20,            "组件数过多"
    illegal = [c for c in components if c not in component_set]
    assert not illegal, f"包含非法组件: {illegal}"
```

**第二层：字段完整性校验**

```python
REQUIRED_FIELDS = ["任务类型", "任务地点"]

def validate_fields(item):
    missing = [f for f in REQUIRED_FIELDS if f not in item["input"]]
    assert not missing, f"缺少必填字段: {missing}"
    assert item["input"]["任务类型"] in VALID_TASK_TYPES, "任务类型非法"
```

**第三层：语义一致性校验（LLM 裁判，抽检用）**

```
判断以下描述和拓扑是否语义一致：
描述：{input}
拓扑：{output}
返回：{"consistent": true/false, "reason": "..."}
```

### 3.5 模型微调方案

| 阶段 | 硬件 | 配置 |
|------|------|------|
| **训练** | A100 40/80G | Qwen2.5-7B-Instruct + LoRA(r=16) + bf16 |
| **导出** | A100 | merge_and_unload → GGUF Q4_K_M 量化 |
| **推理** | RTX 5070 8G | llama.cpp / ollama，显存占用约 4G |
| **约束层** | CPU | Outlines 结构化解码，强制输出合法组件 |

**LoRA 关键参数**：

```python
LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
    lora_dropout=0.05,
    task_type="CAUSAL_LM"
)
```

**推理约束解码**（防止生成非法组件）：

```python
# 使用 Outlines 库，基于 JSON Schema 约束输出
TOPOLOGY_SCHEMA = {
    "type": "string",
    "pattern": "^起飞( -> [\\u4e00-\\u9fa5a-zA-Z0-9]+)+ -> 降落$"
}
generator = outlines.generate.regex(model, pattern)
```

## 七、关键原则

> **数据质量 > 数据数量**
> Week 2 的人工审核是整个方案最重要的环节。种子数据的质量决定了后续 LLM 扩增的上限。宁可花更多时间打磨 100 条高质量种子，也不要用有问题的模板批量生成几千条噪声数据。

> **组件库是边界**
> 所有生成数据的输出拓扑必须严格限制在组件库范围内。模型学的是"任务描述到已有组件的映射"，而不是发明新组件。

> **校验层是保险**
> 即使微调效果很好，生产环境中也必须保留基于组件库的约束解码层，作为最后一道防线，确保系统输出的拓扑100%合法可执行。
